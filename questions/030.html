030. Wzorce Projektowe: Repository i Unit Of Work. Przykłady implementacji w wybranych frameworkach.

📚
💡 Aha! Moment – Repository i Unit of Work to „Biblioteka z systemem wypożyczeń”:

Wyobraź sobie staroświecką bibliotekę. Repository to półki z książkami – możesz poprosić bibliotekarza: „znajdź mi wszystkie książki Tolkiena” albo „dodaj ten nowy egzemplarz na półkę z fantastyką”. Nie musisz wiedzieć, jak książki są ułożone, w którym dokładnie regale stoją, ani jak działa system katalogowania. Dostajesz to, czego potrzebujesz, przez proste, zrozumiałe API. Unit of Work to rejestr wypożyczeń przy ladzie – bibliotekarz zapisuje w nim każdą zmianę: tę książkę wypożyczono, tamtą zwrócono, a ta została zniszczona i trzeba ją wycofać. Na koniec dnia jednym podpisem zatwierdza wszystkie zmiany naraz (commit), zamiast biegać do magazynu po każdej pojedynczej operacji. Jeśli coś pójdzie nie tak, rejestr jest cały unieważniany (rollback) i wszystko wraca do poprzedniego stanu.

  • 🔵 Repository – abstrakcja nad kolekcją obiektów. Ukrywa szczegóły dostępu do danych (SQL, pliki, API). Daje metody: znajdź, dodaj, usuń, filtruj.
  • 🟠 Unit of Work (UoW) – śledzi wszystkie zmiany dokonane na obiektach w ramach jednej operacji biznesowej. Zatwierdza je atomowo (wszystkie naraz) lub wycofuje.
  • 🟢 Razem – Repository daje dostęp do danych, UoW zapewnia spójność transakcyjną. Współpracują, by oddzielić logikę biznesową od mechaniki bazy danych.
  • 🟣 W frameworkach – często są już wbudowane. Hibernate Session to UoW. DbContext w Entity Framework to UoW z wbudowanymi repozytoriami (DbSet). W SQLAlchemy Session to też UoW.

1. Wzorzec Repository – „Półka z książkami, nie szuflada z SQL”

Repository to wzorzec, który pośredniczy między warstwą domeny (logiką biznesową) a warstwą mapowania danych (ORM, surowy SQL, API zewnętrzne). Zachowuje się jak kolekcja obiektów w pamięci: możesz zapytać repo.Find(czyTolkien), dodać repo.Add(nowaKsiazka), usunąć repo.Remove(staraKsiazka). Nie musisz wiedzieć, czy dane są w PostgreSQL, MongoDB, czy w pliku XML – dla Ciebie to po prostu kolekcja.

Kluczowa zasada: Repository udostępnia metody, które odpowiadają językowi domeny, a nie technicznemu SQL-owi. Zamiast SELECT * FROM books WHERE author_id = 123 masz repo.FindByAutor(autor). To sprawia, że logika biznesowa jest czysta, czytelna i testowalna – w testach podmieniasz prawdziwe repozytorium na mocka.

🔑 Generyczne repozytorium: Często tworzy się IRepository<T> z podstawowymi operacjami (GetById, GetAll, Add, Remove), a potem dziedziczy dla konkretnych encji (IBookRepository : IRepository<Book>), dodając specjalistyczne zapytania.
2. Unit of Work – „Rejestr wypożyczeń bibliotekarza”

Unit of Work (UoW) to wzorzec, który śledzi wszystkie zmiany dokonane na obiektach w trakcie jednej operacji biznesowej. Jego zadaniem jest zapewnienie atomowości: albo wszystkie zmiany zostaną zapisane w bazie naraz, albo żadna. UoW rejestruje nowe obiekty (INSERT), zmodyfikowane (UPDATE) i usunięte (DELETE), a przy wywołaniu commit() / saveChanges() generuje odpowiednie zapytania w jednej transakcji. Jeśli coś pójdzie nie tak, transakcja jest wycofywana (rollback). UoW często zawiera też Identity Map – gwarantuje, że każdy wiersz bazy jest reprezentowany przez dokładnie jeden obiekt w pamięci w ramach sesji.

3. Jak Repository i Unit of Work działają razem?

Repository i Unit of Work są projektowane do współpracy. Repository zapewnia dostęp do encji (znajdowanie, dodawanie, usuwanie), a Unit of Work śledzi zmiany dokonane na tych encjach i zatwierdza je transakcyjnie. W praktyce UoW jest często wstrzykiwane do repozytorium, albo repozytoria są tworzone z sesji UoW. Przykład w kodzie:

var klient = klientRepo.FindById(123);
klient.Nazwa = "Nowa nazwa";
zamowienieRepo.Add(noweZamowienie);
uow.Commit(); // zapisuje UPDATE klienta i INSERT zamówienia w jednej transakcji

Kluczowa zaleta: logika biznesowa nie woła bezpośrednio SaveChanges() po każdej zmianie. Gromadzi zmiany i zatwierdza je zbiorczo na końcu operacji. To eliminuje problem częściowych zapisów i zwiększa wydajność (mniej transakcji).

4. Implementacja w frameworkach – „Wbudowane wsparcie”
☕ Hibernate / JPA (Java)
Session to Unit of Work. Śledzi zmiany (dirty checking), zarządza transakcjami. EntityManager (JPA) działa jak UoW z repozytoriami (find, persist, remove). Możesz tworzyć własne repozytoria opakowujące EntityManager. Spring Data JPA automatycznie generuje repozytoria z metod nazwanych (findByNazwisko).
🔷 Entity Framework Core (.NET)
DbContext to Unit of Work. Śledzi zmiany i zapisuje je przez SaveChanges(). DbSet<T> to gotowe repozytorium z metodami Add, Remove, Find, LINQ. Możesz tworzyć własne repozytoria opakowujące DbContext dla lepszej testowalności.
🐍 SQLAlchemy (Python)
Session to Unit of Work. Metody add(), delete(), query() to operacje repozytoryjne. session.commit() zatwierdza wszystkie zmiany. session.rollback() wycofuje. Możesz tworzyć klasy repozytoriów przyjmujące Session w konstruktorze.
5. Przykład implementacji (C# / Entity Framework Core)
// === Interfejs generycznego repozytorium ===
public interface IRepository<T> where T : class
{
  T GetById(int id);
  IEnumerable<T> GetAll();
  void Add(T entity);
  void Remove(T entity);
}

// === Unit of Work ===
public interface IUnitOfWork : IDisposable
{
  IRepository<Klient> Klienci { get; }
  IRepository<Zamowienie> Zamowienia { get; }
  int Commit(); // SaveChanges()
}

// === Użycie w serwisie ===
public void ZlozZamowienie(int klientId, Zamowienie z)
{
  var klient = uow.Klienci.GetById(klientId);
  klient.DodajZamowienie(z);
  uow.Zamowienia.Add(z);
  uow.Commit(); // transakcja: UPDATE klient + INSERT zamowienie
}
⚠️ Pułapki Egzaminacyjne:
  • Repository to nie tylko DAO! DAO (Data Access Object) jest niskopoziomowe, często zwraca surowe dane. Repository operuje na encjach i mówi językiem domeny.
  • Nie każda metoda w repozytorium potrzebuje własnego interfejsu. Jeśli używasz EF Core, DbSet już jest repozytorium. Dodatkowe opakowanie ma sens dla testowalności i izolacji od frameworka.
  • SaveChanges() za często = problem. Wołanie Commit() po każdej zmianie zabija ideę Unit of Work. Zbieraj zmiany i zatwierdzaj je razem.
  • Identity Map – pamiętaj o niej. W ramach jednego UoW ta sama encja (ten sam ID) to ten sam obiekt. Jeśli zmodyfikujesz go przez jedno repozytorium, zmiana będzie widoczna we wszystkich innych. To dobrze, ale trzeba być tego świadomym.
📋 Egzaminacyjny Cheat-Sheet – Repository i Unit of Work:
Wzorzec Rola W EF Core / Hibernate
Repository Abstrakcja nad kolekcją obiektów. Metody: Find, Add, Remove, Query. EF: DbSet<T>. Hibernate: EntityManager.find().
Unit of Work Śledzi zmiany, zarządza transakcją. Atomowy commit lub rollback. EF: DbContext.SaveChanges(). Hibernate: Session.commit().
