065. Interfejs programowania aplikacji (API).

🍽️
💡 Aha! Moment – „Kelner, który łączy Cię z kuchnią":

Wyobraź sobie restaurację. Siedzisz przy stoliku i masz przed sobą menu – listę dań, które możesz zamówić. Nie wchodzisz do kuchni, nie grzebiesz w garnkach, nie krzyczysz do kucharza. Mówisz kelnerowi: „Poproszę pizzę Margherita i wodę mineralną". Kelner przekazuje zamówienie do kuchni, a po chwili wraca z Twoim daniem. Co dokładnie dzieje się w kuchni – jakich składników użyto, na jakiej patelni smażono – Cię nie interesuje. Ważne, że dostałeś to, co zamówiłeś.

API (Application Programming Interface) działa identycznie. To „kelner" w świecie oprogramowania – zestaw reguł, który mówi: „Możesz poprosić o te rzeczy (endpoints), w taki sposób (metody HTTP, parametry), a ja dostarczę Ci odpowiedź (JSON, XML)". Nie musisz wiedzieć, jak system działa w środku – czy używa PostgreSQL czy MongoDB, czy kod jest w Javie czy Pythonie. Wysyłasz żądanie, dostajesz odpowiedź. API to menu, kelner i drzwi do kuchni w jednym.

🏷️ Kryteria oceny i Słowa kluczowe (Active Recall): API (Application Programming Interface) REST (Representational State Transfer) GraphQL SOAP gRPC Endpoint / Zasób (Resource) HTTP: GET, POST, PUT, DELETE, PATCH JSON / XML – formaty odpowiedzi Autoryzacja: API Key, OAuth 2.0, JWT Rate Limiting, Paginacja, Wersjonowanie
1. API – umowa między systemami

API to zbiór zdefiniowanych reguł i protokołów, które umożliwiają komunikację między różnymi aplikacjami. API mówi: „Jeśli wyślesz mi żądanie w takim formacie, odpowiem Ci w takim formacie, wykonując przy tym konkretną operację". To kontrakt – obie strony (klient i serwer) go znają i przestrzegają.

API nie jest nowym wynalazkiem – korzystasz z niego codziennie. Gdy aplikacja pogodowa na telefonie pokazuje temperaturę, łączy się przez API z serwerem meteorologicznym. Gdy logujesz się przez Google na jakiejś stronie, strona ta komunikuje się z API Google. Gdy płacisz kartą w sklepie, terminal rozmawia z API banku. API jest dosłownie wszędzie.

🗣️ Jak to powiedzieć: „API to umowa między systemami – zestaw reguł, który mówi, jak wysłać żądanie i w jakim formacie otrzymamy odpowiedź. To jak menu w restauracji – nie wchodzimy do kuchni, tylko składamy zamówienie przez kelnera. API jest wszędzie: aplikacja pogodowa łączy się z serwerem meteorologicznym, logowanie przez Google to komunikacja z API Google, a płatność kartą to rozmowa terminala z API banku. Najważniejsze: API ukrywa implementację – nie musimy wiedzieć, czy serwer używa PostgreSQL czy MongoDB, Java czy Pythona. Wysyłamy żądanie, dostajemy odpowiedź."

Nie ma jednego najlepszego – wybór zależy od tego, czy priorytetem jest szybkość, elastyczność, czy bezpieczeństwo. Proszę spojrzeć na planszę."
3. REST w praktyce – HTTP, zasoby, metody

Ponieważ REST dominuje w nowoczesnych aplikacjach webowych, przyjrzyjmy się mu bliżej. REST opiera się na trzech filarach:

  • Zasoby – wszystko jest zasobem (np. /users, /orders/123). URL-e są rzeczownikami, nigdy czasownikami. Źle: /getUsers. Dobrze: GET /users.
  • Metody HTTP – GET (pobierz), POST (utwórz), PUT (podmień całość), PATCH (zmień część), DELETE (usuń).
  • Reprezentacje – klient dostaje dane w formacie, który rozumie (JSON, XML), a nie bezpośredni dostęp do bazy.
🗣️ Jak to powiedzieć: „REST opiera się na trzech filarach. Po pierwsze, zasoby – URL-e są rzeczownikami, nigdy czasownikami. Nie piszemy getUsers, tylko GET /users. Po drugie, metody HTTP – GET do pobierania, POST do tworzenia, PUT do podmiany całości, PATCH do częściowej zmiany, DELETE do usuwania. Po trzecie, reprezentacje – klient dostaje dane w JSON lub XML, a nie bezpośredni dostęp do bazy. Proszę spojrzeć na przykłady na planszy – pokazują one kompletne API CRUD dla zasobu Książki z metodami, endpointami i przykładową odpowiedzią JSON."
4. Bezpieczeństwo API – zamek na drzwiach kuchni

Publiczne API bez zabezpieczeń to jak kuchnia bez drzwi – każdy może wejść i narobić szkód. Standardowe mechanizmy ochrony:

  • API Key – prosty klucz w nagłówku. Dobry dla serwisów B2B, ale nie dla użytkowników końcowych.
  • OAuth 2.0 + JWT – standard branżowy. Użytkownik loguje się, dostaje token, a każdy request zawiera go w nagłówku Authorization: Bearer eyJhbG.... Token wygasa, można go odświeżyć.
  • Rate Limiting – ogranicza liczbę zapytań na minutę. Chroni przed przeciążeniem (DDoS) i nadużyciami.
🗣️ Jak to powiedzieć: „Bezpieczeństwo API to osobny, kluczowy temat. API Key to najprostsza metoda – klucz w nagłówku, dobry dla komunikacji między firmami. Standardem branżowym jest OAuth 2.0 z JWT – użytkownik loguje się, dostaje token, który wysyła w każdym żądaniu w nagłówku Authorization Bearer. Token wygasa i można go odświeżyć, co zwiększa bezpieczeństwo. Rate limiting ogranicza liczbę zapytań na minutę, chroniąc przed atakami DDoS i nadużyciami. Bez tych mechanizmów publiczne API to jak kuchnia bez drzwi – każdy może wejść."
⚠️ Pułapki Egzaminacyjne (Grzechy API):
  • „GET do modyfikacji danych" – Kardynalny błąd. GET /api/deleteUser/5 narusza zasady REST i bezpieczeństwa. GET musi być idempotentny i bezpieczny (bez efektów ubocznych). Do usuwania służy DELETE.
  • „API = REST" – REST to najpopularniejszy styl, ale nie jedyny. GraphQL, gRPC, SOAP, WebSockets też są API. Wybór zależy od potrzeb.
  • „Wersjonowanie API jest zbędne" – Gdy zmienisz strukturę odpowiedzi JSON (np. usuniesz pole), wszystkie aplikacje klienckie, które z niej korzystały, przestają działać. Wersjonuj przez URL (/v1/...) lub nagłówek Accept.
🗣️ Jak odpowiedzieć komisji na pytanie o pułapki: „Trzy największe pułapki przy projektowaniu API. Po pierwsze, używanie GET do modyfikacji danych – GET musi być bezpieczny i bez efektów ubocznych, do usuwania służy DELETE. Po drugie, mylenie API z REST-em – GraphQL, gRPC i SOAP też są API. Po trzecie, brak wersjonowania – gdy zmienimy strukturę JSON, wszystkie aplikacje klienckie przestają działać. Wersjonujemy przez URL z prefixem v1 lub przez nagłówek Accept."

🧠 Grokking means truly understanding.
API to nie tylko technologia – to język, którym systemy rozmawiają ze sobą. Dobre API jest jak dobrze napisane menu: zrozumiałe, przewidywalne i nie wymaga zaglądania do kuchni. Gdy projektujesz API, myśl o programiście, który będzie z niego korzystał za rok – bo tym programistą możesz być Ty sam.
