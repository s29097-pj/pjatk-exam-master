018. Deklarowanie i definiowanie funkcji.

🏭
💡 Aha! Moment – Deklaracja to „Plakat reklamowy”, definicja to „Hala produkcyjna”:

Wyobraź sobie, że funkcja to usługa oferowana przez fabrykę kodu. Deklaracja (zwana też prototypem) to plakat przed fabryką – mówi przechodniom (kompilatorowi): „Tu możesz zamówić usługę o nazwie X, potrzebujesz takich materiałów (parametry), a dostaniesz taki produkt (typ zwracany)”. Definicja to wnętrze fabryki – rzeczywiste maszyny i robotnicy (ciało funkcji), którzy wykonują pracę. Plakat nie wystarczy, żeby wyprodukować towar – potrzebujesz hali. W językach takich jak C i C++ to rozdzielenie jest fundamentalne, bo pozwala na ogromne projekty składające się z setek plików.

  • 🔵 Deklaracja (prototyp) – mówi kompilatorowi: „Taka funkcja istnieje”. Zawiera nazwę, typ zwracany i typy parametrów.
  • 🟠 Definicja – zawiera rzeczywiste ciało funkcji (kod do wykonania). Każda definicja jest jednocześnie deklaracją, ale nie odwrotnie.
  • 🟢 Pliki nagłówkowe (.h) – katalogi usług pełne deklaracji. Pliki źródłowe (.c) – hale produkcyjne z definicjami.
  • 🟣 Zasada pojedynczej definicji (ODR) – możesz rozwiesić wiele identycznych plakatów (deklaracji), ale halę produkcyjną (definicję) musisz mieć dokładnie jedną.

1. Deklaracja vs Definicja – „Plakat i fabryka”

W C/C++ oddzielenie deklaracji od definicji jest świadomym wyborem projektowym, który umożliwia kompilację wieloplikową i ukrywanie implementacji. Deklaracja to obietnica, że funkcja istnieje – definicja to jej spełnienie.

🗣️ Jak to powiedzieć: „Proszę Państwa, kluczowe jest rozróżnienie deklaracji i definicji. Deklaracja, zwana prototypem, to plakat informacyjny dla kompilatora – mówi, że istnieje funkcja o danej nazwie, przyjmująca określone typy parametrów i zwracająca konkretny typ. Definicja to rzeczywiste ciało funkcji, czyli hala produkcyjna z kodem. W C i C++ możecie mieć wiele deklaracji – jak wiele kopii tego samego plakatu – ale definicja musi być dokładnie jedna w całym programie. To jest zasada ODR, One Definition Rule. Proszę spojrzeć na planszę: deklaracja kończy się średnikiem i nie ma nawiasów klamrowych, definicja ma ciało w klamrach.”
2. Struktura wieloplikowa – „Katalog usług i hale produkcyjne”

W profesjonalnych projektach C/C++ deklaracje funkcji (oraz definicje typów, makr, stałych) umieszcza się w plikach nagłówkowych (.h). Definicje funkcji trafiają do plików źródłowych (.c, .cpp). Inne pliki dołączają nagłówki przez #include, co daje im dostęp do obietnic. Linker na końcu łączy wszystkie hale w jeden spójny program.

🗣️ Jak to powiedzieć: „W praktyce w C i C++ deklaracje lądują w plikach nagłówkowych z rozszerzeniem .h, a definicje w plikach źródłowych .c. Plik nagłówkowy to jak katalog usług – każdy, kto go dołączy przez include, wie, jakie funkcje są dostępne. Plik źródłowy to hala produkcyjna z rzeczywistym kodem. Na końcu linker scala wszystkie hale w jeden program wykonywalny. Kluczowa jest ochrona nagłówków przed wielokrotnym dołączaniem – używamy ifndef, define i endif. Bez tego kompilator zgłosi błąd wielokrotnej definicji.”
3. Parametry – „Materiały do produkcji”

Parametry formalne (w deklaracji/definicji) to nazwy zastępcze dla danych, które funkcja otrzyma. Parametry aktualne (argumenty) to konkretne wartości przekazane przy wywołaniu. Niektóre języki (C++, Python) pozwalają na argumenty domyślne – jeśli klient nie poda materiału, fabryka użyje standardowego.

Domyślne (C++, Python)
int potega(int x, int wykl=2) { … }
Wywołanie: potega(5) → x=5, wykl=2. potega(5,3) → x=5, wykl=3.
Przeciążanie (C++, Java)
int max(int a, int b);
double max(double a, double b);
Ta sama nazwa, różne typy/liczba parametrów. Kompilator wybiera właściwą wersję.
🗣️ Jak to powiedzieć: „Parametry formalne to zmienne zastępcze w definicji funkcji – mówią, jakie materiały fabryka przyjmie. Parametry aktualne, czyli argumenty, to konkretne wartości przy wywołaniu. W C++ i Pythonie możemy podać argumenty domyślne – jeśli klient ich nie dostarczy, fabryka użyje standardowych wartości. Uwaga: w C++ argumenty domyślne muszą być na końcu listy parametrów. Kolejna cecha C++ i Javy to przeciążanie funkcji – możemy mieć kilka funkcji o tej samej nazwie, różniących się typami lub liczbą parametrów. Kompilator wybiera odpowiednią na podstawie sygnatury.”
4. Jak to wygląda w innych językach?
🐍 Python
Brak oddzielnych deklaracji. def nazwa(param): jest jednocześnie deklaracją i definicją. Wszystko jest dynamiczne. Moduły przez import.
☕ Java
Wszystko jest metodą w klasie. Nagłówek metody to deklaracja. Ciało w { } to definicja. Interfejsy (interface) zawierają wyłącznie deklaracje.
⚙️ C
Ścisłe rozdzielenie deklaracji (.h) i definicji (.c). Konieczność ochrony przed wielokrotnym dołączaniem (#ifndef).
🗣️ Jak to powiedzieć: „Warto porównać, jak różne języki podchodzą do deklaracji i definicji. C wymaga ścisłego rozdzielenia – deklaracje w .h, definicje w .c. Python w ogóle nie ma oddzielnych deklaracji – słowo kluczowe def od razu tworzy funkcję, która jest obiektem. Java stoi pośrodku: metody mają nagłówki, które można umieszczać w interfejsach jako czyste deklaracje, a definicje są w klasach. Te różnice wynikają z filozofii języka: C daje pełną kontrolę i umożliwia kompilację przyrostową, Python stawia na szybkość pisania, a Java na kontrakty i architekturę.”
5. Jak działa wywołanie funkcji? – Stos wywołań

Przy wywołaniu funkcji tworzona jest ramka stosu zawierająca parametry aktualne, adres powrotu i zmienne lokalne. Gdy funkcja kończy działanie, ramka jest niszczona, a sterowanie wraca do miejsca wywołania.

🗣️ Jak to powiedzieć: „Na koniec warto wspomnieć o stosie wywołań, bo to mechanizm ściśle związany z funkcjami. Przy każdym wywołaniu funkcji na stosie tworzona jest ramka – zawiera argumenty, adres powrotu i zmienne lokalne. Gdy funkcja kończy działanie, ramka jest zdejmowana ze stosu. Dzięki temu funkcje mogą wywoływać się nawzajem, a po zakończeniu sterowanie wraca dokładnie tam, skąd przyszło. To elegancki mechanizm, ale jeśli funkcja wywołuje samą siebie bez końca – czyli nieskończona rekurencja – stos się przepełni i program się wysypie.”
⚠️ Pułapki Egzaminacyjne:
  • Deklaracja ≠ definicja. W C możesz zadeklarować funkcję wiele razy, ale zdefiniować tylko raz. Brak definicji = błąd linkera.
  • W C deklaracja bez parametrów to (void), nie puste nawiasy! int foo(); oznacza „nieokreślona liczba parametrów”, nie „zero parametrów”.
  • Nagłówki bez strażników: #ifndef MATH_H / #define MATH_H … #endif zapobiega wielokrotnemu dołączaniu.
  • Argumenty domyślne w C++ tylko od prawej! int f(int a, int b=2, int c) jest błędem. Domyślne muszą być na końcu.
🗣️ Jak odpowiedzieć komisji na pytanie o pułapki: „Największe pułapki dotyczą C. Po pierwsze, deklaracja z pustymi nawiasami, jak int foo(), nie oznacza funkcji bezparametrowej – to stara konwencja mówiąca o nieokreślonej liczbie parametrów. Poprawnie powinno być int foo(void). Po drugie, brak strażników w nagłówkach grozi wielokrotnym dołączeniem i błędami kompilacji. Po trzecie, w C++ argumenty domyślne muszą być deklarowane od prawej strony – int f(int a, int b=2, int c) jest błędem składniowym. I najważniejsze: zapamiętajcie zasadę ODR – wiele deklaracji, dokładnie jedna definicja w całym programie.”

🧠 Grokking means truly understanding.
Deklaracja to obietnica – mówisz kompilatorowi „zaufaj mi, ta funkcja istnieje”. Definicja to dowód – dostarczasz rzeczywistą implementację. Rozdzielając je, budujesz kod jak z klocków: wymienny, testowalny i czytelny.
